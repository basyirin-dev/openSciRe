# openscire-sandbox — Sandboxed Computational Discovery with Reproducibility-First Architecture

## I. Overview & Motivation

openscire-sandbox is OpenSciRe's "Does it work?" engine — a sandboxed, secure, reproducible computational experimentation platform that executes code generated by the hypothesis system, measures outcomes across multiple transparent criteria, and returns everything (winners, losers, and the environmental cost) to the provenance DAG. It is the structural answer to Google's AlphaEvolve + ERA, rebuilt from the epistemic ground up.

Why does this component exist? Google's Computational Discovery stack (AlphaEvolve for evolutionary code optimization, ERA for parallel scoring and "Empirical Research Assistance") was published in *Nature* and represents the most ambitious attempt to automate computational science. But as detailed in [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), it is structurally unsound across every dimension that matters for real science. It commits an optimization-discovery category error — mistaking hill-climbing on a pre-defined fitness function for open-ended scientific discovery. It lacks a falsification architecture, generating confirmation castles instead of testable claims. It runs arbitrary code at scale with no structural security model. It scores its own outputs in a self-scoring circular trust loop. It burns massive compute without carbon accounting. It cherry-picks winners and discards the graveyard of failures that contains the most scientific information. And it produces unreproducible findings by failing to capture the environmental provenance — dependencies, seeds, hardware — that would allow another lab to verify the result. These are not bugs; they are *architectural commitments* to a verificationist, brute-force, confidence-over-humility model of computational science.

The core inversion that openscire-sandbox represents is threefold. First, it is built for falsification-capable execution — meaning every computational experiment is designed to be proven wrong, not just scored high. The FalsificationAgent in openscire-hypothesis uses a different objective function than the DiscoveryAgent, and openscire-sandbox executes both, recording every negative result as a first-class provenance artifact. Second, it is reproducibility-first, meaning the execution environment is the star of the show: pinned dependencies, captured random seeds, hardware profiling, and signed reproducibility bundles are not bolted on after the fact but designed into the execution pipeline. Third, it is carbon-aware and local-first — no Google-scale cluster required, no hidden environmental externalities, no computational apartheid that restricts discovery to those with enterprise cloud budgets (see [01_overview.md](../../critiques/01_overview.md), Gap 5 — enterprise pricing cliff, and [02_philosophy.md](../../critiques/02_philosophy.md), §8 — environmental philosophy).

How does it fit into the OpenSciRe ecosystem? openscire-sandbox receives experiments from openscire-hypothesis (both discovery proposals and falsification attacks), executes them in an isolated Rust sandbox with full environment capture, and returns structured results — including null results, negative results, and verification failures — to the provenance DAG. It also contextualizes results against existing knowledge via openscire-literature and passes through the EthicalFirewall and CarbonBudgetTracker from the ethics layer ([03_ethics_layer.md](../../docs/phases/03_ethics_layer.md)). Every execution is a provenance event: signed, timestamped, parent-chained, and auditable. The sandbox does not decide what is true; it reports what happened, under what conditions, and at what cost — and leaves the epistemology to the human.

## II. Sandbox Architecture

### A. Rust Sandbox Core (PyO3)

The sandbox core is written in Rust and exposed to the Python-based OpenSciRe ecosystem via PyO3 bindings. Rust was chosen for four reasons that directly address the gaps identified in the AlphaEvolve critique [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md):

1. **Memory safety without garbage collection** — Critical for a code execution engine that runs LLM-generated code. GC pauses are unacceptable when the sandbox must enforce hard execution deadlines. Rust's ownership model guarantees no use-after-free, no buffer overflows, and no dangling pointers, eliminating entire classes of memory-safety vulnerabilities that plague sandbox implementations in C/C++.

2. **Strong concurrency guarantees** — The sandbox must handle multiple simultaneous executions (user-initiated, hypothesis-driven, falsification-driven). Rust's `Send` and `Sync` traits prevent data races at compile time, eliminating race conditions that could allow escape from the sandbox or corruption of execution state.

3. **Minimal runtime** — Rust compiles to native code with no VM overhead and minimal runtime. This means the sandbox startup time is dominated by process creation, not language runtime initialization. Target: <50ms startup overhead for simple executions.

4. **seccomp-bpf system call filtering** — Rust's libc bindings and the `seccompiler` crate allow compile-time construction of seccomp-bpf (Berkeley Packet Filter) programs that whitelist only the system calls a given execution environment needs. This is the kernel-level enforcement mechanism that prevents sandbox escape, network access, and file system exfiltration.

**PyO3 Bridge Architecture:**

```rust
// openscire-sandbox-core/src/lib.rs — PyO3 binding surface

use pyo3::prelude::*;

#[pyclass]
pub struct Sandbox {
    config: SandboxConfig,
    seccomp_filter: SeccompProfile,
}

#[pymethods]
impl Sandbox {
    #[new]
    fn new(config: SandboxConfig) -> PyResult<Self> { ... }

    fn execute(&self, code: &str, language: &str) -> PyResult<ExecutionResult> { ... }

    fn dry_run(&self, code: &str, language: &str) -> PyResult<StaticAnalysisReport> { ... }

    fn generate_reproducibility_bundle(&self, execution_id: &str) -> PyResult<ReproducibilityBundle> { ... }
}
```

The Python call path is: `openscire-core` agent → calls `Sandbox.execute()` via PyO3 → Rust performs static analysis → spawns isolated process with seccomp restrictions → captures stdout/stderr/exit code/resources → verifies output integrity → returns `ExecutionResult` to Python. The entire lifecycle is synchronous from the Python perspective but the Rust layer manages process lifecycle, timeout enforcement, and resource tracking internally.

**Filesystem restrictions:** Each execution receives an ephemeral `tmpfs` workspace. No access to the host filesystem except a configurable allowlist (e.g., shared data directories, output directories). The workspace is destroyed after result capture. This prevents the "arbitrary code execution risk" identified in Gap 13 of [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md).

**Network restrictions:** By default, no external network access. A network allowlist can be configured (e.g., for API calls to specific databases), enforced at the seccomp level via `socket()` and `connect()` filtering and at the application level via a SOCKS proxy that only allows traffic to allowlisted hosts. This addresses Gap 13's network exfiltration concern.

**Memory limits:** Configurable per-execution budget (default: 2GB). Enforced via cgroups v2 memory controller. Hard limit — `SIGKILL` if exceeded, with allocation tracking in the result.

**Execution timeout:** Configurable hard kill (default: 300s). The Rust layer sends `SIGKILL` to the process group at timeout, not `SIGTERM` (which can be caught and ignored by malicious code).

**System call filtering:** seccomp-bpf profile generated at sandbox creation time based on the requested execution language and capabilities. For a standard Python execution, the profile whitelists approximately 60 syscalls (read, write, mmap, brk, clone, etc.) and blocks approximately 300 (mount, reboot, kexec_load, etc.). This is the last line of defense: even if the Rust layer has a vulnerability, the kernel enforces restrictions.

**Process isolation:** Each execution runs in a separate child process with its own PID namespace. The Rust parent process monitors the child and collects exit status. Process groups prevent orphaned child processes from lingering after timeout. This prevents fork bombs (Gap 13 — resource exhaustion) and ensures clean state between executions.

### B. Execution Environment

The execution environment is designed to solve the reproducibility crisis at the source. The key insight from [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), Gap 14 (Dependency and Environment Fragility), is that a computational "discovery" that only works on a specific Google TPU cluster with a specific JAX version is not a scientific discovery — it is a platform-specific artifact. openscire-sandbox treats the environment as a first-class object:

**Environment lockfile:** Before any execution, the sandbox captures or generates a complete lockfile of the execution environment. For Python executions, this is a `pip freeze` snapshot with package versions, hashes, and source URLs. The lockfile is part of the reproducibility bundle.

**Dependency tree recording:** Beyond the flat lockfile, the sandbox recursively records the dependency tree — which packages depend on which, with version constraints — allowing exact reconstruction of the environment. This is stored in a machine-readable format (JSON-LD with RO-Crate compatibility).

**Config snapshot:** The full `SandboxConfig` — time limit, memory limit, network allowlist, filesystem mounts, environment variables, language version — is serialized and attached to every execution result. This is digitally signed and included in the reproducibility bundle.

**Random seed management:** If the executed code uses random number generation, the sandbox captures the seed from the Python-side `random.seed()` or `numpy.random.seed()`. If no seed is set, the sandbox auto-seeds with a captured seed and records it. Every execution becomes deterministic if replayed with the same seed bundle.

**Data hash verification:** Input data files are hashed (SHA-256) before execution. The hash is stored in the execution provenance. If the same experiment is re-run with the same input data, hash verification confirms bit-for-bit identical inputs. If data has changed, the sandbox flags the reproducibility impact.

**Hardware profile capture:** CPU model, core count, RAM, GPU model and VRAM, OS kernel version, and glibc version are captured and stored. This enables the crucial epistemic check: "Did this result depend on specific hardware?" If a model scores well on an A100 but poorly on a consumer GPU, that is a finding, not a bug. Hardwired into the bundle rather than an afterthought, this solves Gap 14's core complaint: the exact environment is preserved.

### C. Code Security

Security in openscire-sandbox operates at three layers, mimicking the defense-in-depth of mature sandbox systems (nsjail, Firecracker) but specialized for LLM-generated scientific code.

**Pre-execution static analysis:** Before any code touches a runtime, the sandbox's static analyzer inspects the AST for:
- Dangerous imports (`os`, `subprocess`, `socket`, `ctypes`, `syscalls`)
- Filesystem access patterns (writing outside the workspace)
- Network calls (`urllib`, `requests`, `http.client`)
- System introspection (`platform.uname`, `os.uname`)
- Fork/exec patterns
- Infinite loop potential (heuristic: nested loops without bounds)

The static analyzer returns a `StaticAnalysisReport` with risk flags. Configurable action on detection: warn (continue with annotation), block (refuse execution), or flag (log and continue). This is the EthicalFirewall for code. Integration with the risk tier system ([03_ethics_layer.md](../../docs/phases/03_ethics_layer.md)) means Tier 1 (high risk) domains trigger mandatory block on dangerous patterns, while Tier 3 (low risk) domains may allow with warning.

**Post-execution output sanitization:** Raw stdout/stderr is captured but not immediately released. The output sanitizer strips:
- File paths that could leak host system structure
- Environment variable dumps containing secrets
- Stack traces that reveal sandbox internals
- Binary data that could contain exploits

The sanitized output is validated against the allowed schema (structured JSON, plain text, CSV, etc.) before release to the caller.

**No persistent shell access from within sandbox:** The sandbox environment is ephemeral by design. There is no network-accessible shell, no SSH daemon, no reverse tunnel capability. Even if code inside the sandbox achieves code execution, it is inside a blank tmpfs with no network, no persistence, and a ticking timeout clock.

**Structured channels:** All output goes through `stdout → capture → validate → release`. There is no direct pipe from the sandboxed process to the caller's stdout. The Rust layer intermediates every byte. This prevents terminal escape sequences, ANSI injection, and other output-based attacks.

## III. Discovery Engine

### A. Multi-Criteria Evaluation (vs Single Fitness Function)

The single most dangerous architectural choice in AlphaEvolve is the fitness function — a scalar value that determines which code variation "wins." As [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), Gap 7 (Fitness Function Tyranny) and Gap 24 (Quantifiability Bias) detail: a single fitness function embeds a hidden value judgment, cannot handle contested or unknown metrics, and structurally excludes problems that resist scalar scoring. A climate model optimized for temperature accuracy may silently fail on precipitation, ocean acidification, or biodiversity cascades. The tool amplifies the user's blind spots at scale.

openscire-sandbox uses multi-criteria evaluation with full transparency:

- **User-configurable evaluation criteria** — The researcher specifies *multiple* objective functions, each with a weight and a direction (maximize, minimize, target). These are not hidden — they are displayed alongside results so the researcher can inspect tradeoffs.
- **Pareto front reporting** — Instead of a single "winner," the sandbox reports the Pareto frontier of non-dominated solutions. A solution that is worse on accuracy but dramatically faster, or consumes 10x less energy, is reported as a tradeoff, not a loser.
- **Criterion provenance** — Every evaluation criterion is recorded with its source: did the user define it? Was it suggested by the hypothesis system? Does it encode a known physical constraint (conservation law, boundary condition)? This prevents zombie metrics from silently shaping results.
- **Multiple objective evaluation in parallel** — The sandbox can execute the same code against different evaluation harnesses simultaneously, producing a multi-dimensional score matrix rather than a single number.
- **Inspectable and modifiable** — After seeing the Pareto frontier, the user can adjust criteria weights and re-score without re-executing (if the raw outputs are sufficient), or re-execute with new criteria. The evaluation layer is not a black box.

This directly addresses Gaps 7, 24, and 45 (Aesthetic/Elegance — Gap 45 notes that preference for simpler models with fewer parameters is a legitimate criterion that scalar fitness ignores). By making criteria transparent and multiple, the sandbox does not impose a single value system on scientific judgment.

### B. Null Hypothesis Generation (Deferred — Post-Pilot)

A rigorous computational study requires comparison against baseline models that encode "no effect" or "random behavior." As [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), Gap 10 (Absence of Null Model Generation) and Gap 28 (no conceptual-experimental feedback loop) detail, AlphaEvolve's score improvements are uninterpretable without null baselines. Does the new model capture disease dynamics, or does it have more parameters that fit noise?

Null hypothesis generation in openscire-sandbox is deferred to post-pilot because it requires significant domain modeling that cannot be generically applied. The planned approach:

- **Null models**: For a given problem, generate the simplest possible model (mean predictor, linear baseline) and random-parameter models.
- **Randomization tests**: Shuffle the labels/targets and re-run to establish the null distribution of scores.
- **Baseline comparison**: Every execution result is compared against the null distribution, reporting a p-value-like metric: "How likely would this score arise from random chance?"

Deferred because the automated generation of theoretically justified null models requires domain-specific knowledge (what constitutes "no effect" varies by field). The infrastructure for null comparison exists in the reproducibility bundle and multi-criteria evaluation framework; the null generation itself waits on openscire-hypothesis's NullHypothesisAgent (see [09_sandbox.md](../../docs/phases/09_sandbox.md), Task 9.4 — deferral documentation).

### C. Reproducibility Bundle

The ReproducibilityBundle is the sandbox's answer to [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), Gap 17 (Reproducibility Path Problem) and Gap 20 (Absence of Independent Replication Architecture). AlphaEvolve archives the winner but not the tournament; openscire-sandbox archives everything needed to replay the execution verbatim.

The bundle contains:

- **Environment lockfile** — Complete pinned package versions with hashes (Pipfile.lock, conda-lock, Cargo.lock)
- **Dependency tree** — Recursive dependency graph with version constraints
- **Config snapshot** — Complete `SandboxConfig` serialized to JSON
- **Random seeds** — All captured seeds for PRNG, numpy, torch, tensorflow
- **Data hashes** — SHA-256 hashes of all input data files
- **Hardware profile** — CPU, GPU, RAM, OS, kernel, glibc
- **All inputs and parameters** — Code source, CLI arguments, environment variables
- **All outputs** — stdout, stderr, exit code, execution time, resource usage
- **Signed with Ed25519** — The bundle is cryptographically signed with the project key, enabling third-party verification that the bundle has not been tampered with

The bundle is exportable in multiple formats: JSON (for programmatic use), RO-Crate (for repository deposit), and W3C PROV (for provenance tracking). This enables the independent replication architecture that Gap 20 demands. A lab in another institution can download a reproducibility bundle, verify the signature, and re-run the exact experiment on their hardware — and if the result differs, that is a finding.

### D. Negative Result Integration

Science's file-drawer problem — the structural bias toward publishing positive results while negative results rot in filing cabinets — is amplified to industrial scale by systems that discard 9,999 losing variations and only surface the winner. [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), Gap 22 (Cherry-Picking of Winners) identifies this as survivorship bias; Gap 48 (Inability to Fail Productively) notes that the graveyard of failed code is a lost archive of negative knowledge.

openscire-sandbox records **every** execution attempt as a provenance event:

- **ALL execution attempts recorded** — Not just successful or high-scoring ones. Every invocation of `Sandbox.execute()` produces a provenance entry with status, score, and metadata. There is no "discard low scores" pipeline; there is only a provenance graph.
- **NegativeResultRegistry integration** — Failed experiments (timeouts, crashes, low scores, falsification successes) are sent to the shared NegativeResultRegistry (openscire-core). This is not a junk drawer; it is a structured, searchable database of what did not work, tagged with domain, technique, and reason for failure.
- **"Graveyard" browser** — The registry is explorable: "Show me all failed attempts to model solar flare prediction using fluid dynamics between 2024 and 2026." A failed approach from 2024 may become viable with new algorithms or data — the registry ensures that negative knowledge is preserved, not discarded.
- **Cross-linking** — When a user or agent proposes an experiment, the system checks the graveyard: "Has anyone already tried this?" If yes, the sandbox surfaces the previous attempt, its failure mode, and the conditions under which it failed, before the new execution begins. This prevents wasted compute on known dead ends and turns the graveyard into a collective intelligence layer.

This architecture directly addresses Gaps 22, 48, and also Gap 10 from [03_structural_triad.md](../../critiques/03_structural_triad.md) (Missing Negative Result & Null Hypothesis Architecture), which identifies that the triad's structural optimism — built to generate positive findings — is one of science's deepest structural flaws.

## IV. Ethical & Environmental Architecture

### A. Carbon Budget Tracker (Phase 3.5)

[05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), Gap 34 (Environmental Cost of Brute-Force Discovery) identifies that running "thousands of variations in parallel" for complex simulations is computationally extravagant — the equivalent of strip-mining for knowledge, externalizing carbon costs while internalizing scientific credit. [02_philosophy.md](../../critiques/02_philosophy.md), §8 (Environmental Philosophy) expands this into a critique of the metabolic rift between compute-intensive AI and ecological sustainability.

The CarbonBudgetTracker, integrated into the sandbox execution pipeline:

- **FLOPs to kWh to CO2e estimation** — Each execution estimates FLOPs from known computational complexity, hardware utilization (captured in hardware profile), and execution time. FLOPs → kWh conversion uses hardware-specific TDP and utilization factors. kWh → CO2e uses configurable grid carbon intensity (user provides their region or accepts defaults).
- **Per-query carbon report** — Every execution result includes a carbon report displayed in CLI/UI output and logged in provenance: "This execution consumed 0.023 kWh, estimated equivalent to 0.15 km of driving."
- **Configurable monthly carbon budget** — User sets a monthly carbon budget in kg CO2e. The sandbox tracks cumulative usage. Warning at 80%. **Hard stop at 100%** — the sandbox refuses to execute until the user explicitly overrides (logged and justified in provenance). This is the structural "stop button" that Gap 51 (Absence of a "Stop" Button) demands.
- **Local estimation without cloud profiling** — When hardware-specific data is unavailable (no GPU, unknown CPU TDP), the tracker uses conservative defaults and flags the estimate as uncertain. Better an uncertain estimate than no accounting at all.

### B. Dual-Use Code Detection

[05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), Gap 35 (Dual-Use and Harmful Code Generation Risk) notes that unlike biological dual-use research, code is immediately executable. An agent generating thousands of code variations could produce optimized algorithms for surveillance, bioweapon dispersal, or financial exploitation, with no ethical review gate between generation and execution.

The EthicalFirewall ([03_ethics_layer.md](../../docs/phases/03_ethics_layer.md)) operates at both the prompt level (experiment request) and response level (generated code) of the sandbox:

- **DURC pattern detection in code** — Pattern matching and embedding-based classification detect dual-use research of concern categories: pathogen enhancement, toxin synthesis, weapons delivery, AI safety evasion, surveillance hardening. Detection operates locally (no cloud dependency) using a bundled classifier.
- **Configurable action** — Flag (log only), warn (user visible message), block (prevent execution), escalate (notify admin). The action depends on the risk tier classification of the research domain.
- **Integration with Risk Tier** — Tier 1 domains (virology, toxin research, dual-use chemistry) trigger mandatory block on DURC code patterns. Tier 3 domains (solar forecasting, mathematics) may proceed with warning. The tier system, detailed in [03_ethics_layer.md](../../docs/phases/03_ethics_layer.md), §3.2, applies to computational experiments as much as to hypothesis generation.
- **False positive feedback loop** — Users can contest flags; contested flags are reviewed and used to tune the classifier. All firewall decisions are written as non-removable, append-only provenance entries.

### C. Fitness Function Transparency

[05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), Gap 19 (Self-Scoring Circular Trust Problem) identifies that AlphaEvolve generates variations and scores them without an independent validation layer. When the scorer and the generator are the same system, the result is structurally self-certifying — a student grading their own exam.

openscire-sandbox's evaluation criteria are transparent by design:

- **Evaluation criteria are NOT hidden** — The multi-criteria evaluation function is displayed in the execution result, with weights, formulae, and sources. There is no "secret sauce" scoring.
- **User can inspect, modify, and compare criteria** — The criteria configuration is part of the SandboxConfig, which is versioned, signed, and stored in provenance. If a user modifies criteria between executions, the change is tracked.
- **Multiple criteria shown simultaneously** — The Pareto front visualization shows all criteria simultaneously. A model that sacrifices accuracy for energy efficiency is visible as a tradeoff, not a failure.
- **VerificationAsymmetryTracker integration** — Every generated claim (including evaluation scores) is categorized by verifiability: fully verifiable (reproducible from captured seeds and data), partially verifiable (requires additional experiments), or non-verifiable (currently untestable). This breaks the self-scoring loop by making verification status explicit, not assumed.

### A. Core Data Models (Pydantic)

    input_code: str
    input_parameters: Dict[str, Any]

### B. Rust-Python Bridge

The PyO3 bridge exposes three sandbox objects to the Python layer:

1. **`Sandbox.new(config: SandboxConfig)`** — Create a sandbox instance with a given configuration. The Rust side compiles the seccomp filter, prepares the tmpfs workspace, and validates the config. Returns a handle.

2. **`sandbox.execute(code: str, language: str) -> ExecutionResult`** — The primary entry point. Performs: (a) static analysis of the code, (b) pre-flight checks against EthicalFirewall, (c) carbon budget check, (d) process creation with seccomp, (e) execution with timeout and memory monitoring, (f) result capture, (g) output sanitization, (h) carbon accounting, (i) provenance entry generation.

3. **`sandbox.dry_run(code: str, language: str) -> StaticAnalysisReport`** — Static analysis only, no execution. Used by the EthicalFirewall to pre-screen code before the user decides to execute.

**Error handling and timeout management:**

All sandbox errors are mapped to typed error codes:

```rust
// openscire-sandbox-core/src/errors.rs

#[derive(Debug, thiserror::Error)]
pub enum SandboxError {
    #[error("Static analysis blocked execution: {0}")]
    BlockedByStaticAnalysis(String),

    #[error("Execution exceeded time limit of {0}s")]
    Timeout(u64),

    #[error("Execution exceeded memory limit of {0}MB")]
    MemoryOverflow(u64),

    #[error("Seccomp violation: blocked syscall {0}")]
    SeccompViolation(String),

    #[error("Internal sandbox error: {0}")]
    Internal(String),

    #[error("Carbon budget exhausted")]
    CarbonBudgetExhausted,

    #[error("Ethical firewall blocked: {0}")]
    EthicalFirewallBlock(String),

    #[error("Environment setup failed: {0}")]
    EnvironmentError(String),
}
```

Theta errors propagate through PyO3 as Python exceptions with the same hierarchy, ensuring that the Python layer can handle sandbox failures programmatically.

### C. Integration with OpenSciRe Core

openscire-sandbox is not an isolated component — it is threaded into every other subsystem:

**openscire-hypothesis integration:** The hypothesis system submits experiments to the sandbox as structured `SandboxExecution` requests. Each request includes the code to execute, the evaluation criteria, the expected null baseline, and the falsification objective (if applicable). The sandbox executes and returns results to hypothesis for the next iteration of the discovery/falsification cycle. The NegativeResultRegistry is shared: when the FalsificationAgent succeeds in refuting a hypothesis, that negative result is archived alongside the original discovery attempt.

**openscire-literature integration:** Execution results are contextualized against existing literature. If the sandbox reports a result that contradicts established literature, the system flags this. If the sandbox result confirms a finding that was previously only theoretical, the system notes this. The provenance DAG links execution results to the literature references that informed the hypothesis, creating an end-to-end evidentiary chain.

**Provenance DAG integration:** Every execution — successful, failed, blocked, timed out — is a `ProvenanceEntry` in the DAG. The entry includes: `action_type` ("sandbox_execute"), `agent_id` (which agent requested the execution), `model_id` (which model provided the code), `parameters_snapshot` (the SandboxConfig), `input_hash` (code hash), `output_hash` (result hash), `parent_ids` (linking to the hypothesis that generated the code), timestamp, and Ed25519 cryptographic signature. This enables the complete tournament replay that AlphaEvolve lacks (Gap 17).

**EthicalFirewall integration:** The EthicalFirewall operates at two points in the sandbox pipeline: (1) before execution, scanning the prompt (experiment description) and the code (static analysis + DURC pattern detection), and (2) after execution, scanning the output for dangerous content. Firewall decisions are recorded as non-removable provenance entries. Integration with Risk Tier Classification means the severity of the firewall action varies by domain.

### vs. Google AlphaEvolve + ERA

| Evaluation transparency | Hidden scoring function | Inspectable, modifiable criteria |
| Human override | No structural stop button | Mandatory override gate before execution |
| Attribution | Orphan discoveries | Ed25519-signed provenance chains |

### vs. DeepMind AlphaFold / AlphaTensor

DeepMind's domain-specific tools (AlphaFold for protein folding, AlphaTensor for matrix multiplication optimization) are genuine scientific breakthroughs within their narrow domains. They do not claim to be general-purpose computational discovery engines. openscire-sandbox is fundamentally different in kind:

- **General-purpose vs. domain-specific**: AlphaFold is a protein structure prediction tool. AlphaTensor optimizes matrix multiplication. openscire-sandbox executes arbitrary computational experiments in any domain expressible in code.
- **Domain-expert tool vs. epistemic platform**: AlphaFold is used *by* structural biologists. openscire-sandbox is used *to build* and *evaluate* computational models. It is meta-scientific infrastructure, not domain science.
- **Black-box vs. transparent**: AlphaFold's internal scoring is opaque even to its users. openscire-sandbox's evaluation criteria are inspectable, modifiable, and auditable.
- **Reproducibility**: AlphaFold results depend on model weights that are updated and versioned; exact reproduction across versions is not guaranteed. openscire-sandbox's ReproducibilityBundle guarantees bit-for-bit reproducibility.

### vs. Traditional Code Sandboxes (Docker, nsjail, Firecracker)

Traditional sandbox technologies solve the security problem but not the scientific reproducibility problem. openscire-sandbox is different in three ways:

- **Reproducibility as a first-class feature**: Docker containers capture the environment, but they do not capture random seeds, do not hash input data, do not profile hardware, and do not sign execution bundles. openscire-sandbox treats these as structural requirements, not optional layers.
- **Scientific-code-specific static analysis**: Standard sandboxes analyze code for security (SQL injection, shell injection). openscire-sandbox also analyzes for scientific risks: overfitting (multiple comparisons), dual-use patterns, conservation law violations, and data leakage.
- **Evaluation harness integration**: Docker provides an execution environment and nothing more. openscire-sandbox provides the execution environment *plus* the multi-criteria evaluation harness, the carbon tracker, the negative result archiver, and the provenance DAG integration — all as a single, coherent abstraction.

## IX. Limitations Acknowledged

Consistent with the project's epistemic stance (see [AGENTS.md](../../AGENTS.md), §1 — rigorous, irreverent, fiduciary), the following limitations are explicitly acknowledged:

1. **Null hypothesis generation deferred to post-pilot.** The sandbox infrastructure supports null comparison (baselines, randomization tests), but the automated generation of theoretically justified null models — which require domain knowledge — is deferred. Until this is implemented, null baselines must be provided by the user or the hypothesis agent.

2. **Physical/wet lab validation remains out of scope.** openscire-sandbox operates entirely *in silico*. It cannot design a clinical trial, pipette a sample, or validate a model against physical reality. The sandbox explicitly labels all outputs as "computational experiments, not empirical validation" and leaves the bridge to material science to human researchers. This is a structural constraint, not an engineering oversight. As [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), Gap 21 (Physical Validation Chasm) notes, that chasm is real and cannot be papered over by UI.

3. **Full symbolic-numeric integration deferred.** The sandbox can execute Python code that calls SymPy, SageMath, or Mathematica (via API), but it has no native symbolic reasoning engine. It cannot derive a closed-form solution, prove stability conditions, or verify conservation laws symbolically unless the user explicitly includes these checks in the code. [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), Gap 18 (Lack of Symbolic-Numeric Integration) identifies this as a gap that is acknowledged and deferred.

4. **Maximum computation scale limited by local hardware.** openscire-sandbox is designed for commodity hardware (laptop, workstation, modest server). It does not support Google-scale parallel execution of thousands of variations across TPU clusters. For problems that require massive parallel compute, the sandbox can submit to a Ray/Dask cluster (planned), but this is not the primary use case. The tradeoff is intentional: local-first enables accessibility (Gap 38 — Digital Divide) while constraining maximum throughput.

5. **No guarantee of correctness for generated code.** The sandbox validates that code is *safe* to execute (seccomp, static analysis) and that results are *reproducible* (bundle, seeds, hashes). It does not validate that code is *scientifically correct*. A model that encodes a subtle physical error but runs safely and reproduces consistently will pass both checks. This is not a bug; it is an epistemic boundary. The sandbox reports what happened; the human decides what it means.

6. **No adversarial red-teaming of evaluation criteria.** The multi-criteria evaluation system is transparent and modifiable, but it is not automatically tested for gaming or manipulation. A sufficiently sophisticated researcher could craft code that scores well on user-specified criteria while being scientifically hollow. This is an acknowledged vulnerability that mirrors the real-world problem of Goodhart's law in scientific metrics.

7. **Does not solve the attribution and credit problem.** Ed25519-signed provenance chains provide attribution *within* the system (which agent requested which execution), but they do not solve the broader attribution problem identified in [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md), Gap 33: who is the author of a discovery made with computational assistance? openscire-sandbox provides the data for attribution decisions but does not impose a policy.

8. **Does not handle the "winter/summer" temporal calibration automatically.** The sandbox captures temporal context (the data timestamp, the seasonal window) but does not automatically detect that a model trained on summer data will fail in winter. Users must specify temporal validation splits. Automated temporal drift detection is deferred.

These limitations are not admissions of failure; they are the boundaries that make the sandbox an honest tool rather than a dishonest oracle. A tool that claims to do everything is trusted with nothing. openscire-sandbox claims to do sandboxed, reproducible, carbon-aware, falsification-capable computational experimentation — and is designed to do those specific things rigorously, transparently, and with epistemic humility.

## X. Synthesis: The North Star

openscire-sandbox is not "AlphaEvolve but open source." It is a fundamental inversion of the computational discovery architecture. Where AlphaEvolve optimizes and scores, openscire-sandbox executes and reports. Where AlphaEvolve discards failures, openscire-sandbox archives them as collective intelligence. Where AlphaEvolve hides its scoring function, openscire-sandbox displays evaluation criteria as inspectable, modifiable first-class objects. Where AlphaEvolve requires Google-scale compute, openscire-sandbox runs on a laptop and tells you the carbon cost.

The sandbox's single most critical design choice is that it does **not** claim to discover anything. It executes code, captures the environment, measures outcomes against transparent criteria, records failures, accounts for carbon, signs everything with a cryptographic key, and hands the results to a human. The discovery — if any — happens in the human's interpretation, informed by openscire-hypothesis's agents and openscire-literature's context but executed in the irreducible space of human judgment that no algorithm can fill.

This is the answer to the synthesis of [05_alphaevolve_era.md](../../critiques/05_alphaevolve_era.md): the problem with AlphaEvolve is not that it is a bad optimizer, but that it is an optimizer pretending to be a scientist. openscire-sandbox is not a scientist. It is a tool for scientists — honest about its limits, rigorous in its reporting, and designed to serve the human process of discovery rather than replace it.
