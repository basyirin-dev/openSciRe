# openscire-hypothesis — Multi-Agent Hypothesis Generation with Falsification-First Architecture

## I. Overview & Motivation

openscire-hypothesis is OpenSciRe's "What if?" engine — a multi-agent hypothesis generation system that takes literature-synthesized questions, generates candidate hypotheses through genuinely diverse agent deliberation, stress-tests them via dedicated falsification, filters them through an epistemic safety layer, and registers both surviving and failed hypotheses in a persistent provenance graph. It is the falsification-first replacement for Google's Co-Scientist, designed from first principles to address not only the technical gaps but the epistemological, ethical, and structural failures inherent in the Gemini for Science approach.

Google's Co-Scientist (Nature 2025) simulates the scientific method via multi-agent "idea tournaments" where homogeneous LLM agents generate, debate, and evaluate hypotheses. On paper, this sounds like a step toward automated scientific reasoning. In practice, it is a **monologue disguised as dialogue** — agents sharing the same base model, the same training data, the same architectural inductive biases, engaged in a tournament that optimizes for rhetorical fluency rather than empirical fidelity. The critiques are extensive and damning: no falsification engine [04_co_scientist.md](../../critiques/04_co_scientist.md#II.7), no experimental design reasoning [04_co_scientist.md](../../critiques/04_co_scientist.md#II.8), no negative result architecture [04_co_scientist.md](../../critiques/04_co_scientist.md#I.3), no epistemic humility calibration [04_co_scientist.md](../../critiques/04_co_scientist.md#I.5), no dual-use firewall [04_co_scientist.md](../../critiques/04_co_scientist.md#VII.31), and no mechanism for genuine cognitive diversity [04_co_scientist.md](../../critiques/04_co_scientist.md#III.11). It is a hypothesis generator optimized for normal science within existing paradigms — exactly the kind of tool that accelerates puzzle-solving while systematically missing the revolutionary breaks that transform understanding [02_philosophy.md](../../critiques/02_philosophy.md#II.7).

The core inversion that defines openscire-hypothesis is **falsification-first architecture**. Where Co-Scientist optimizes for generating plausible, citation-supported hypotheses that survive an internal debate, openscire-hypothesis structurally privileges the search for *what disproves a hypothesis*. Every hypothesis generated by the system is immediately routed to a dedicated FalsificationAgent whose sole purpose is to destroy it — to find the confound, the counter-example, the methodological flaw, the contradictory evidence. This is not a debate stage in a tournament; it is an architectural hard stop. A hypothesis that survives falsification is more trustworthy; one that does not is registered as a first-class NegativeResult in the persistent registry, contributing to the collective knowledge of what does not work.

A second core inversion is **genuine agent diversity**. Where Co-Scientist deploys clones of the same model in a simulated debate, openscire-hypothesis enforces architectural heterogeneity: agents can use different LLM providers (local Ollama, vLLM, OpenAI-compatible BYOK, Anthropic), different temperature and prompt strategies, different context sources, and — critically — different objective functions. The FalsificationAgent's goal is the opposite of the HypothesisGenerator's goal. This is not a polite committee; it is an adversarial epistemology engine that mirrors the actual social structure of science: the skeptic, the enthusiast, the methodologist, the ethicist, and the domain specialist, each with different cognitive commitments [02_philosophy.md](../../critiques/02_philosophy.md#II.1).

A third inversion is **negative results as first-class citizens**. Co-Scientist has no structural mechanism for capturing what failed — hypotheses that lose the tournament are discarded, their existence and rationale invisible to the user [04_co_scientist.md](../../critiques/04_co_scientist.md#IV.20). openscire-hypothesis treats every failed hypothesis as a scientific asset. The NegativeResultRegistry persists all falsified claims, null findings, and failed experimental designs with structured metadata, cross-linking to related literature, and searchable provenance. Over time, this registry becomes a map of the field's silent failures — the negative knowledge that human scientists accumulate through experience but that AI systems, trained on positive-result-biased corpora, structurally lack [04_co_scientist.md](../../critiques/04_co_scientist.md#I.3).

**How it fits into OpenSciRe**: openscire-hypothesis receives literature insights and gap analyses from openscire-literature (Phase 4/5), which provides the multi-source synthesized context (PubMed, arXiv, Semantic Scholar, OpenAlex, Zotero, CNKI, SciELO) that forms the epistemic ground for hypothesis generation. Generated hypotheses that pass falsification and ethics review are routed to openscire-sandbox for computational testing (Phase 8), and the results of those tests feed back into both the literature engine and the hypothesis generator, closing the recursive loop that Google's triad structurally lacks [03_structural_triad.md](../../critiques/03_structural_triad.md#II.1). The entire lifecycle — from literature question through hypothesis generation through falsification through experimental design through computational test through result registration — is tracked in a provenance DAG (W3C PROV / RO-Crate), ensuring that every claim, every decision, and every failure is auditable and reproducible.

## II. Agent Architecture

### A. Core Agents (MVP — Phase 6)

The MVP of openscire-hypothesis deploys four specialist agents orchestrated by a SupervisorAgent, each with heterogeneous architecture, distinct objective functions, and independent epistemic commitments. This is not a committee meeting; it is a structured adversarial process designed to surface the truth through disciplined conflict.

#### 1. SupervisorAgent

The SupervisorAgent is the orchestration state machine and the human-interface governor. It does not generate scientific content; it manages the process by which content is generated, tested, and validated.

**Orchestration state machine**: The SupervisorAgent transitions through a defined lifecycle: `idle → planning → executing → reviewing → completed → failed`. Each state has defined entry criteria, exit criteria, timeout thresholds, and escalation paths. If the system hangs in `executing` for longer than the configured timeout for a task, the SupervisorAgent declares `failed` and initiates diagnostic logging.

**Research plan decomposition**: Given a research question or problem statement, the SupervisorAgent decomposes it into subtasks dispatchable to specialist agents. For example, "Does gene X regulate pathway Y in disease Z?" decomposes into: a literature review subtask (what is known about X, Y, Z), a hypothesis generation subtask (propose mechanistic models), a falsification subtask (find evidence against each model), and an ethics subtask (screen for dual-use or consent issues).

**Task queue management**: The SupervisorAgent maintains a priority queue of tasks, dispatches them to available agents, tracks completion status, and handles retries on timeout or error. Task dependencies are respected — falsification does not begin until a hypothesis exists; ethics review does not begin until a falsification report exists.

**Agent health monitoring**: Each agent reports heartbeat at configurable intervals. The SupervisorAgent detects stalled or crashed agents, restarts them, and re-dispatches orphaned tasks. If an agent fails repeatedly (configurable threshold, default 3), the SupervisorAgent escalates to human.

**Conflict resolution**: When specialist agents disagree — e.g., the HypothesisGenerator proposes mechanism A, the LiteratureReviewAgent finds contradictory evidence, the FalsificationAgent finds a confound — the SupervisorAgent manages an adjudication workflow. This may involve: requesting additional literature from the LiteratureReviewAgent, asking the FalsificationAgent to strengthen its case, escalating to a human arbiter (for Tier 1/2 ethical gates), or registering the conflict as an open question in the provenance graph.

**Human handoff**: At architectural hard stops (Tier 1/2 ethical gates, resource constraints beyond configured budgets, unresolved agent conflicts), the SupervisorAgent pauses the workflow, packages the current state with full provenance, and routes it to the human interface. The workflow cannot proceed without explicit human sign-off at these checkpoints [03_structural_triad.md](../../critiques/03_structural_triad.md#II.14).

**Session persistence**: The SupervisorAgent serializes the complete state of all agents, message queues, task statuses, and provenance graph at configurable intervals. A session can be saved, closed, and restored days or months later — enabling the longitudinal research narratives that Co-Scientist's stateless architecture precludes [04_co_scientist.md](../../critiques/04_co_scientist.md#VIII.39).

#### 2. LiteratureReviewAgent

The LiteratureReviewAgent is not a general-purpose chatbot with access to search. It is a structured evidence-gathering system with specific methodological commitments: it finds what is known, what is contradictory, what is absent, and what is unreliable.

**Query decomposition**: Given a research question, the agent decomposes it into sub-queries optimized for different search engines. A question about "CRISPR off-target effects in neural tissue" generates queries targeting PubMed (clinical), arXiv (computational methods), Semantic Scholar (cross-disciplinary), and domain-specific sources.

**Multi-source search dispatch (Phase 4 engines)**: The agent dispatches queries to the Phase 4 literature engine, which supports: PubMed/MEDLINE, arXiv, Semantic Scholar, OpenAlex, Zotero libraries (user-provided), CNKI (Chinese National Knowledge Infrastructure), and SciELO (Latin American and Caribbean). This directly addresses the linguistic and geographic bias critique [04_co_scientist.md](../../critiques/04_co_scientist.md#VI.26) — openscire-hypothesis is not an English-only system.

**Result deduplication and relevance filtering**: Retrieved results are deduplicated across sources (same paper from PubMed and Semantic Scholar) and filtered by relevance to the original research question using embedding similarity and keyword overlap. Low-relevance results are logged but not discarded — they remain in the provenance graph as "considered but excluded."

**Literature synthesis with citation support**: The agent produces structured summaries of the retrieved literature, organized by theme, with each factual claim tagged to its source document. This synthesis is the input to the HypothesisGenerator and the FalsificationAgent.

**Gap identification**: Beyond summarizing what is known, the agent identifies areas with insufficient literature coverage — the spaces where a hypothesis could make a novel contribution. "Gap" here means: no papers address this specific question, existing studies have contradictory findings, or existing studies use methods that the agent can critique [07_hypothesis_generation.md](../../docs/phases/07_hypothesis_generation.md).

**Contradiction detection**: The agent actively compares findings across sources and flags contradictions — two papers claiming opposite results, a replication that failed, a meta-analysis that undermines the consensus view. These contradictions are surfaced to the HypothesisGenerator as raw material for alternative hypotheses [04_co_scientist.md](../../critiques/04_co_scientist.md#II.9].

**Evidence quality assessment**: Every source is rated on methodology quality (RCT vs. observational vs. preprint), sample size, citation impact, retraction status (via RetractionWatch integration), and source venue reputation. This rating is attached to every claim derived from the source and is preserved through the entire hypothesis lifecycle. The SourceEnforcer (Section V.C) uses these ratings to determine whether a cited claim is actually supported by its source.

#### 3. FalsificationAgent (THE KEY DIFFERENTIATOR)

The FalsificationAgent is openscire-hypothesis's most important architectural innovation — a dedicated adversarial agent whose sole purpose is to destroy hypotheses. Where every other agent in the ecosystem is optimizing for generation, synthesis, or construction, the FalsificationAgent optimizes for refutation. It is the structural embodiment of Popperian epistemology, and its existence is the single clearest difference between openscire-hypothesis and Google's Co-Scientist, which has no dedicated agent for falsification [04_co_scientist.md](../../critiques/04_co_scientist.md#II.7).

**Popperian falsification search**: Given a hypothesis, the FalsificationAgent actively searches for evidence that contradicts it — not evidence that confirms it, not evidence that is ambiguous, but evidence that, if valid, would logically refute the claim. It queries the literature engine with searches designed to surface disconfirming results: "X does NOT cause Y in Z," "failure to replicate mechanism A," "contradictory evidence for pathway B."

**Counter-example generation**: The agent constructs concrete scenarios where the hypothesis would fail — edge cases, boundary conditions, parameter regimes where the proposed mechanism should break down. For a hypothesis about drug efficacy, it asks: "For which patient populations would this drug be ineffective? For which dosages would toxicity outweigh benefit? Under which environmental conditions would the mechanism fail?"

**Experimental confound identification**: The agent analyzes the hypothesis for uncontrolled variables — factors that could produce the predicted outcome through a mechanism other than the one proposed. It produces a structured confound table: confound, direction of bias, severity, proposed control.

**Alternative explanation generation**: For any phenomenon that the hypothesis explains, the FalsificationAgent proposes at least two competing explanations — Occam's razor alternatives, simpler models, known mechanisms that could produce the same observation. This directly counters the hypothesis-first bias [04_co_scientist.md](../../critiques/04_co_scientist.md#II.6] by ensuring that no hypothesis is evaluated in isolation.

**Methodology critique**: The agent evaluates whether the experimental designs proposed by the ExperimentalDesignAdvisor (Section IV.B) actually test the hypothesis. If a proposed experiment cannot distinguish between the hypothesis and an alternative explanation, the methodology is flagged as inconclusive. A hypothesis for which no discriminating experiment can be designed is flagged as currently untestable.

**Falsification report**: The agent produces a structured document containing: the original hypothesis, all falsification attempts made (both successful and unsuccessful), the results of each attempt, remaining uncertainty (what has not been tested), and specific suggestions for strengthening the hypothesis. This report becomes part of the hypothesis's permanent record and influences its confidence score.

#### 4. EthicsAgent

The EthicsAgent is openscire-hypothesis's structural conscience — a dedicated agent that evaluates every hypothesis, research direction, and experimental design against ethical frameworks before any work proceeds. It is not an afterthought or a compliance checkbox; it is an architectural gate that can stop a research workflow.

**Query EthicalFirewall for all hypotheses**: Every hypothesis generated by the system is routed through the EthicalFirewall (Phase 3.1), which scans for dual-use research of concern (DURC) patterns using keyword + embedding-based detection. The detection is fully local, with no cloud dependency [04_co_scientist.md](../../critiques/04_co_scientist.md#VII.31].

**Tier classification**: The agent classifies each research context into one of three risk tiers based on domain, methodology, and data sources. Tier 1 (high risk: virology, toxin research, dual-use chemistry, human genetic engineering) triggers mandatory 24-hour cooling-off and requires external human authorization. Tier 2 (medium risk: clinical research without IRB, human subjects data, animal research) triggers mandatory human notification and optional pause. Tier 3 (low risk: standard computational research) proceeds with standard oversight [03_ethics_layer.md](../../docs/phases/03_ethics_layer.md).

**Dual-use flagging**: Beyond keyword matching, the agent uses embedding-based semantic scanning to detect dual-use potential in research questions and proposed experiments — not just explicit DURC terms but indirect pathways to harmful applications. All flags are logged in the non-removable, append-only provenance audit trail [03_ethics_layer.md](../../docs/phases/03_ethics_layer.md).

**Data sovereignty check**: Before any data is ingested or generated, the agent verifies that the data sources have appropriate consent and usage rights. Public data is flagged as public; proprietary data is flagged with access restrictions; indigenous knowledge is blocked from ingestion into training or generation pipelines per the CARE Principles (Collective Benefit, Authority to Control, Responsibility, Ethics) [03_ethics_layer.md](../../docs/phases/03_ethics_layer.md).

**Carbon budget check**: The agent estimates the compute cost of proposed experiments (FLOPs estimation, kWh conversion, CO2e based on regional grid carbon intensity) and checks against the user-configured monthly carbon budget. If the proposed workflow would exceed the remaining budget, the agent blocks execution until the user explicitly confirms [03_ethics_layer.md](../../docs/phases/03_ethics_layer.md).

**Ethics report**: A structured document is produced: risks identified, tier classification, flags raised (with explanation), data sovereignty assessment, carbon cost estimate, and recommendations. This report is appended to the provenance graph as a permanent record.

**Escalation**: For Tier 1/2 issues, the EthicsAgent cannot proceed alone — it must pause the workflow and notify the human through the SupervisorAgent's handoff mechanism. This is an architectural hard stop, not a dismissible UI popup [03_structural_triad.md](../../critiques/03_structural_triad.md#II.14].

### B. Deferred Agents (Post-Pilot — Phase 7+)

These agents are designed, specified, and architecturally anticipated but deferred from the MVP to manage implementation complexity. Their interfaces are defined; they integrate into the same AgentBus and SupervisorAgent orchestration as the core agents.

**NullHypothesisAgent**: Generates null models and baseline comparisons for every positive hypothesis. For "Compound X inhibits pathway Y," the NullHypothesisAgent generates "Compound X has no effect on pathway Y" and "The observed effect is explained by experimental artifact Z." This directly counters the missing null hypothesis problem [04_co_scientist.md](../../critiques/04_co_scientist.md#II.10].

**HypothesisCriticAgent**: Structured evaluation of hypotheses using multiple criteria: novelty (how far from dominant paradigm), plausibility (evidence consistency), testability (can it be empirically distinguished?), impact (what changes if true?), and elegance (parsimony, explanatory power). Each criterion is scored independently and transparently, unlike Co-Scientist's hidden single evaluation metric [04_co_scientist.md](../../critiques/04_co_scientist.md#VIII.37).

**MethodologyAdvisorAgent**: Experimental design recommendations at the domain-specific level — appropriate statistical tests for the data type, sample size power analysis, blinding strategies, protocol templates, and resource checklists. This is the structured version of what the ExperimentalDesignAdvisor (Section IV.B) provides at a higher level.

**EvidenceValidatorAgent**: Independent verification of evidence chains — given a hypothesis and its supporting citations, the agent attempts to verify each inferential step. Did paper A actually show what the HypothesisGenerator claims? Does the cited methodology actually support the conclusion? This is a human-reviewer simulation that catches the citation-as-rhetorical-shield problem [04_co_scientist.md](../../critiques/04_co_scientist.md#IV.16].

### C. Agent Diversity Guarantee

The single most damning critique of Co-Scientist's multi-agent architecture is that its agents share the same base model, training data, and inductive biases — making the tournament a **monologue disguised as dialogue** [04_co_scientist.md](../../critiques/04_co_scientist.md#III.11). openscire-hypothesis enforces genuine agent diversity through four mechanisms, each configurable and auditable:

**1. Heterogeneous model providers per agent**: Each agent can be configured to use a different LLM provider — local Ollama (e.g., Qwen2.5-72B for the LiteratureReviewAgent), vLLM (e.g., Llama-3-70B for the HypothesisGenerator), OpenAI-compatible BYOK (e.g., Claude 3.5 Sonnet for the FalsificationAgent), or Anthropic native API (e.g., for the EthicsAgent). This is not optional; the SupervisorAgent validates that no two agents share the same model without explicit user override logged to provenance.

**2. Different temperature/prompt strategies**: Agents are configured with different temperature settings that align with their objective function. The HypothesisGenerator uses higher temperature (0.7-0.9) to encourage novelty and exploration. The FalsificationAgent uses lower temperature (0.1-0.3) for focused, precise counter-example construction. The EthicsAgent uses the lowest setting (0.0-0.1) for consistent, principled classification. Prompt strategies differ structurally: the HypothesisGenerator prompts for "generate plausible mechanisms"; the FalsificationAgent prompts for "find assumptions and break them."

**3. Different objective functions**: This is the deepest diversity guarantee. Agents have structurally opposed goals within the same workflow. The HypothesisGenerator optimizes for generating novel, testable, impactful proposals. The FalsificationAgent optimizes for refuting them. The EthicsAgent optimizes for preventing harm. The LiteratureReviewAgent optimizes for accuracy and completeness. These are not variants of the same optimization problem; they are fundamentally different value functions. No single scoring metric governs all agents.

**4. Different context sources per agent**: Agents receive different context windows, deliberately limited to prevent homogeneous reasoning. The LiteratureReviewAgent gets the full literature synthesis. The HypothesisGenerator gets the gap analysis and contradiction list. The FalsificationAgent gets only the hypothesis and the supporting evidence — its task is to find what the evidence does *not* show. The EthicsAgent gets the hypothesis and experimental design but not the literature, preventing it from being biased by citation authority.

**What this prevents**: The "clone debate" problem where two instances of the same model argue opposite sides but converge on the same reasoning patterns because they share the same priors. When openSciRe's FalsificationAgent uses a different model, different temperature, different prompting strategy, different objective function, and different context from the HypothesisGenerator, their debate is structurally heterogeneous. The disagreement is real, not simulated [02_philosophy.md](../../critiques/02_philosophy.md#II.1].

## III. Tournament & Workflow Design

### A. Tournament Architecture

Where Co-Scientist uses a single multi-agent tournament with a hidden scoring function [04_co_scientist.md](../../critiques/04_co_scientist.md#VIII.37), openscire-hypothesis uses a **phased adversarial workflow** with transparent, multi-criteria evaluation at every stage.

**Hypothesis generation phase**: The LiteratureReviewAgent's synthesis and gap analysis are forwarded to the HypothesisGenerator (a capability within the LiteratureReviewAgent at MVP, a standalone agent post-pilot), which generates candidate hypotheses. Multiple hypotheses are generated per research question — by default 3-5, configurable by the user. Each hypothesis is tagged with supporting evidence, contradictory evidence (from the contradiction detection step), and a preliminary confidence estimate.

**Falsification phase**: Every generated hypothesis is independently routed to the FalsificationAgent. The FalsificationAgent does not receive the HypothesisGenerator's confidence score or supporting evidence — it receives only the hypothesis itself, to avoid anchoring bias. Each hypothesis receives a falsification report.

**Ethics review phase**: Every hypothesis that survives falsification (i.e., the FalsificationAgent cannot find a fatal flaw) is routed to the EthicsAgent for tier classification, dual-use scanning, and carbon budget check. Hypotheses that fail ethics review are not discarded — they are registered in the NegativeResultRegistry with the ethics rationale preserved.

**Evaluation**: Surviving hypotheses are presented to the user with a multi-criteria dashboard: novelty score (how far from existing literature), falsification resistance (how many attempts survived), evidence strength (from LiteratureReviewAgent's quality assessment), testability (from ExpermentalDesignAdvisor), ethical risk tier, and carbon cost. The user can sort, filter, and compare hypotheses on any criterion. There is no single "winner" score — the system surfaces the tradeoffs and lets the human judge.

**Conflict resolution**: When agents disagree — e.g., the FalsificationAgent declares a hypothesis falsified but the human disagrees — the conflict is logged in the provenance graph with both positions, supporting evidence, and the human's rationale. This serves as a training signal and an audit trail. The SupervisorAgent coordinates the resolution workflow, which may include: requesting additional evidence from the LiteratureReviewAgent, soliciting a second opinion (if multiple FalsificationAgent models are configured), or escalating to a domain expert.

### B. Predefined Workflow Templates

Three workflow templates are provided for common scientific scenarios, each mapping a different research epistemology [06_multi_agent.md](../../docs/phases/06_multi_agent.md):

**Template 1: Literature → Gap → Hypothesis → Falsification → Report**
The standard nomothetic workflow. Used when the user has a domain and wants to discover what is not yet known, generate hypotheses to fill the gap, falsify them, and produce a research-ready report. Best for: early-stage research questions, grant proposal preparation, PhD project planning.

**Template 2: Hypothesis → Experimental Design → Ethics → Falsification → Negative Result Registration**
The hypothesis-testing workflow. Used when the user already has a hypothesis and wants to design an experiment, check for ethical issues, falsify, and register both successful and failed outcomes. Best for: existing research programs, replication studies, experimental planning.

**Template 3: Literature Search → Contradiction Detection → Alternative Hypothesis → Review**
The anomaly-driven workflow. Used when the user suspects existing findings are contradictory or incomplete. The LiteratureReviewAgent searches for contradictions, the HypothesisGenerator proposes alternative explanations for the conflicting data, and the FalsificationAgent tests each alternative. Best for: crisis periods in a field, contested findings, paradigm-challenging observations.

### C. Custom Workflow Builder

Users can define custom agent chains programmatically through a YAML or Python configuration DSL:

```yaml
workflow:
  name: "custom-replication-workflow"
  steps:
    - agent: LiteratureReviewAgent
      task: "search:replication_studies, method:meta_analysis"
    - agent: HypothesisGenerator
      task: "generate:3, template:alternative_explanations"
    - agent: FalsificationAgent
      task: "falsify:each, depth:full"
    - agent: EthicsAgent
      task: "screen:all, tier:tier3_bypass_ok"
    - agent: NullHypothesisAgent  # deferred, placeholder
      task: "generate_null:each"
```

The custom workflow builder supports: conditional branching (if falsification succeeds, proceed; if not, register negative result and stop), parallel agent execution (run EthicsAgent and FalsificationAgent simultaneously), iterative loops (re-generate hypothesis after falsification feedback), and human checkpoints at any node.

Workflow status is tracked in real-time: current stage, completed stages, estimated time remaining, bottlenecks identified. Full provenance of every agent interaction is stored in the provenance DAG and is exportable as RO-Crate or W3C PROV.

## IV. Key Capabilities

### A. Serendipity Injection (Phase 7.2)

The most profound scientific discoveries often emerge not from systematic reasoning within a paradigm but from **bisociation** — the collision of two previously unrelated conceptual matrices [02_philosophy.md](../../critiques/02_philosophy.md#II.10]. Co-Scientist has no structural mechanism for cross-domain analogy; its tournaments operate within a user-defined research boundary [04_co_scientist.md](../../critiques/04_co_scientist.md#VI.29). openscire-hypothesis addresses this with the SerendipityInjector.

**Cross-domain analogy**: The agent randomly samples concepts, methods, or findings from unrelated disciplines and tests them for analogical mapping to the user's research question. A cancer researcher studying metastasis might receive a hypothesis derived from fungal mycelium network theory or from urban traffic flow modeling. The connection is not random noise — it is a structured analogical mapping that identifies structural isomorphisms between domains.

**Contradiction-driven exploration**: When the LiteratureReviewAgent detects two sources that disagree, the SerendipityInjector generates hypotheses that attempt to reconcile them — not by averaging the positions but by proposing a third mechanism that explains both sets of data. This turns contradictions from problems into raw material for discovery.

**User-configurable serendipity level**: A single parameter between 0.0 and 1.0 controls exploration intensity. At 0.0, the system generates only conservative, consensus-grounded hypotheses within the immediate literature. At 0.5, cross-domain analogies are drawn from neighboring fields. At 1.0, concepts are sampled from maximally distant domains. The default is 0.2 — enough to spark surprise without overwhelming the user.

**Novelty scoring**: Each serendipity-derived hypothesis is scored on its distance from the dominant paradigm in its field, measured by embedding similarity to the centroid of existing literature. High-novelty hypotheses are flagged as such and presented with the caveat that they are more speculative but potentially more transformative.

**Audit trail**: All serendipitous connections are logged — the source domains sampled, the analogical mapping applied, the randomness seed used (for reproducibility). Science may be serendipitous, but openscire-hypothesis ensures that serendipity is auditable [07_hypothesis_generation.md](../../docs/phases/07_hypothesis_generation.md).

### B. Experimental Design Advisor (Phase 7.3)

Generating a hypothesis is trivial; designing an experiment that can test it is the actual work. Co-Scientist stops at the ideation boundary, producing what amounts to science fiction adorned with citations [04_co_scientist.md](../../critiques/04_co_scientist.md#II.8). The ExperimentalDesignAdvisor bridges this gap.

**Experiment type classification**: The advisor classifies the proposed test into experiment type: computational experiment, wet lab protocol, field study, clinical trial, meta-analysis, or observational study. Each type has different structural requirements that the advisor enforces.

**Control variable identification**: The advisor suggests control conditions, blinding strategies (single-blind, double-blind, triple-blind), randomization approaches (block, stratified, cluster), and sham/placebo controls. It explicitly identifies confounders that the FalsificationAgent flagged and proposes controls for each.

**Sample size estimation**: Power analysis is performed for statistical significance given expected effect size, significance threshold (default α=0.05, configurable), and desired power (default 0.8, configurable). For non-statistical experiments, analogous rigor requirements are proposed (e.g., number of independent replicates in a wet lab, number of interview subjects in qualitative research).

**Statistical method suggestion**: Appropriate statistical tests are recommended based on data type (continuous, categorical, ordinal), distribution (normal, non-parametric), number of groups, and independence assumptions. Effect size measures are suggested. Pre-registration protocols are recommended where applicable.

**Protocol template generation**: A structured experimental protocol is generated with step-by-step instructions, expected duration per step, materials and equipment needed, safety precautions, and stopping criteria. The protocol is exportable as PDF, Markdown, or lab notebook format.

**Reagent/resource checklist**: From the protocol, the advisor generates a checklist of required materials, reagents, equipment, and expertise. Estimated costs per item and total experiment cost are calculated. Availability and alternatives are suggested.

### C. Problem Formulation Advisor (Phase 7.4)

The hardest and most important part of science is asking the right question. Co-Scientist assumes the user already knows what to ask — a dangerous assumption that entrenches poor problem framing and exploits junior researchers who need the most help [04_co_scientist.md](../../critiques/04_co_scientist.md#V.23). The ProblemFormulationAdvisor is openscire-hypothesis's structural intervention.

**Research question framing assistance**: Before any hypothesis generation begins, the advisor helps the user articulate their research question using structured templates (PICO for clinical research, FINER for general science, PO for mechanism-oriented questions). It asks clarifying questions: "What is the population of interest? What is the intervention or exposure? What is the comparison? What is the outcome?"

**Implicit assumption identification**: The advisor analyzes the user's question for hidden assumptions — presuppositions that may not be justified. "Does drug X treat disease Y?" assumes that drug X can be administered to disease Y patients, that disease Y has a measurable outcome, that the mechanism of action is relevant to the pathology. Each assumption is surfaced for vetting before hypothesis generation proceeds.

**Alternative formulations**: The advisor generates alternative framings of the same underlying question. "What are the conditions under which drug X fails to treat disease Y?" — the falsification-centered reframing. "What mechanisms other than drug X's known target could explain its effects?" — the mechanism-agnostic reframing. These alternatives are presented to the user for selection.

**Premature commitment prevention**: If the user's question is narrowly defined (e.g., "Does compound X inhibit enzyme Y in vitro?"), the advisor flags the narrowing and asks whether the broader question ("Does compound X affect pathway Y in vivo?") should be considered first. This prevents the premature foreclosure that characterizes much of hypothesis-driven AI [04_co_scientist.md](../../critiques/04_co_scientist.md#II.6].

### D. Negative Result Registry (Phase 6.6)

The single most important structural gap in Google's ecosystem is the absence of a place for negative results [03_structural_triad.md](../../critiques/03_structural_triad.md#II.10). Co-Scientist discards losing hypotheses; Computational Discovery discards low-scoring code variations. The file drawer problem — science's systematic bias toward publishing positive results — is not only unaddressed but structurally amplified [08_miscellaneous.md](../../critiques/08_miscellaneous.md#XI.36]. openscire-hypothesis's NegativeResultRegistry is the architectural corrective.

**Persistent archive of failed hypotheses**: Every hypothesis that is falsified, fails ethics review, or is rejected by the user is stored permanently in the registry. It is not deleted; it is not archived to cold storage; it remains searchable and cross-linked alongside positive results. The archive is immutable via the provenance DAG's append-only structure.

**Structured metadata**: Each negative result includes: the original hypothesis, the method used to test it, a data summary (what was observed), the result type (null finding, contradictory finding, inconclusive, methodologically flawed), confidence in the negative finding, the specific reason for failure (with citations if available), and suggestions for future work.

**Cross-linking to related literature**: Negative results are automatically cross-linked to the literature that motivated the hypothesis and to any positive results in the same domain. A researcher searching for hypotheses about "gene X in disease Y" will see not only the surviving hypotheses but also "Gene X was tested in disease Y by openscire-hypothesis on 2026-11-15 and the result was negative (reason: insufficient statistical power in the falsification test)."

**"Graveyard" browser**: A dedicated visualization interface allows browsing the registry by domain, hypothesis topic, method, date, or failure reason. The graveyard is not a memento mori — it is a research tool. "Show me all hypotheses about kinase inhibitors that failed due to off-target effects" is a query that the registry supports and that could prevent a researcher from pursuing a direction already known to fail.

**Automated submission**: When the FalsificationAgent successfully falsifies a hypothesis (Section II.A.3), the result is automatically registered. When the EthicsAgent blocks a hypothesis, the rationale is automatically registered. When the user manually rejects a hypothesis, they are prompted to submit a negative result with their reasoning. The default is to register; opting out requires explicit justification.

## V. Epistemic Safety Layer

The Epistemic Safety Layer is not an add-on or a separate module — it is a cross-cutting architectural constraint that every agent, every message, and every output passes through. It is openscire-hypothesis's structural answer to the verification asymmetry crisis [04_co_scientist.md](../../critiques/04_co_scientist.md#I.2] and the citation-as-rhetorical-shield problem [04_co_scientist.md](../../critiques/04_co_scientist.md#IV.16].

### A. Uncertainty Quantifier (Phase 3.6)

Co-Scientist's outputs are dressed in the rhetoric of verification — confident claims supported by clickable citations — but the underlying uncertainty is invisible [04_co_scientist.md](../../critiques/04_co_scientist.md#VIII.38). openscire-hypothesis surfaces uncertainty at every level.

**Confidence scoring for all generated claims**: Every claim, whether generated by the HypothesisGenerator, the LiteratureReviewAgent, or any other agent, is accompanied by a confidence score derived from: source quality (methodology rating, sample size, venue, retraction status), agreement level (corroboration across sources), contradiction level (presence of conflicting evidence), and model certainty signals (logprobs, token probabilities, refusal signals from the underlying LLM).

**Contradiction detection within agent outputs**: The UncertaintyQuantifier monitors agent outputs for internal contradictions — a LiteratureReviewAgent that says "the evidence is strong" but cites only preprints and retracted papers. Contradictions are flagged and surfaced to the user.

**Knowledge boundary flagging**: When the available literature is insufficient to answer a question with confidence, the UncertaintyQuantifier does not allow the system to proceed with hypothesis generation. It flags the question as "currently unanswerable from available literature" and suggests what literature would need to exist to enable a confident response. "I don't know" is a first-class output, not a failure mode.

### B. Verification Asymmetry Tracker (Phase 3.8)

The fundational epistemic problem with LLM-based verification is regression: the verifier shares the same architecture and failure modes as the generator [04_co_scientist.md](../../critiques/04_co_scientist.md#I.2]. The VerificationAsymmetryTracker does not solve this regression (it would require a different cognitive architecture, which only a human can provide), but it makes the asymmetry visible and auditable.

**Claim categorization**: Every claim is categorized as: verifiable (can be tested within available resources — computational experiment, literature cross-check), partially verifiable (testable but expensive, time-consuming, or requiring unavailable expertise), or non-verifiable (currently untestable by principle — requires technology that does not exist, requires infinite computation, requires ethical impossibility). This categorization is always displayed next to the claim.

**Verification status over time**: As new literature arrives and as computational tests are completed, the tracker re-evaluates each claim's verification status. A claim that was non-verifiable in November 2026 might become verifiable in January 2027 when a new dataset is published. The tracker monitors these transitions and alerts the user to previously non-verifiable claims that can now be tested.

**Verification path suggestions**: For verifiable claims, the tracker suggests concrete next steps: "To test this claim, run Experiment X using Dataset Y from source Z, analyze with statistical test W." These suggestions are piped to the ExperimentalDesignAdvisor for protocol generation.

### C. Source Enforcer (Phase 5.5)

The SourceEnforcer is openscire-hypothesis's structural defense against citation-as-rhetorical-shield — the problem where a citation verifies that a paper exists but not that the paper supports the claim, that the paper is correct, or that the paper is relevant [04_co_scientist.md](../../critiques/04_co_scientist.md#IV.16).

**Forced citation enforcement**: Every factual claim in any agent's output — every statement that asserts something about the world — must cite at least one source from the retrieved literature. The SourceEnforcer parses each sentence of agent output, extracts claims, maps them to citations, and flags any claim without a supporting source.

**Source extraction from model responses**: Citations are not assumed to be accurate. The SourceEnforcer parses the cited passage from the source document and compares it to the claim being made. If the citation does not actually support the claim (the paper says something different), the claim is flagged as "uncited" even if a citation string is present.

**Source verification**: The SourceEnforcer checks that every cited source exists in the retrieved literature (not hallucinated), that the source has not been retracted (via RetractionWatch integration), and that the source's evidence quality rating (Section II.A.2) is above a configurable threshold. Sources below threshold are flagged.

**Unsupported claim flagging**: Claims without verified citations are highlighted in red in the output. The hypothesis cannot proceed to the next workflow stage until all claims are either cited or explicitly flagged as speculative (which requires user acknowledgment).

### D. Confabulation Detection

Confabulation — the LLM's tendency to generate plausible-sounding falsehoods — is the primary epistemic hazard of using LLMs for scientific work. openscire-hypothesis detects confabulation through multi-agent cross-validation.

**Cross-agent validation of factual claims**: When an agent makes a factual claim, other agents can independently verify it. If the FalsificationAgent claims "No evidence exists for mechanism X," the LiteratureReviewAgent independently searches and either confirms (no evidence found) or contradicts (evidence exists). The cross-validation result is logged and affects confidence scoring.

**Historical pattern tracking for hallucination-prone topics**: The system maintains a registry of topics and domains where agents have been known to confabulate — based on past verification asymmetry tracking. When a new hypothesis touches a hallucination-prone topic, all claims are automatically downgraded in confidence and flagged for human review.

**Automatic knowledge boundary flagging**: When an agent attempts to make a claim about a domain outside its configured expertise (e.g., the LiteratureReviewAgent speculating about quantum mechanics when its literature corpus is molecular biology), the claim is flagged as crossing a knowledge boundary and is routed to the UncertaintyQuantifier for review.

## VI. Dual-Use & Ethics Architecture

### A. DURC Detection

The EthicalFirewall (Phase 3.1) scans every hypothesis, research question, and experimental design for dual-use research of concern patterns. The scanning is fully local — no cloud dependency, no data exfiltration risk.

**Detection methods**: Two-layer scanning. First, keyword-based matching against DURC category definitions (pathogen enhancement, toxin synthesis, weapons delivery, AI safety evasion, surveillance hardening, chemical/biological weapons precursors). Second, embedding-based semantic scanning that detects dual-use potential even when no explicit DURC keywords are present — a hypothesis about "optimizing viral envelope protein stability" would be flagged even if the phrase "gain of function" never appears.

**Configurable action levels**: On detection, the firewall can: flag (log only, no user visible change), warn (user visible warning, workflow continues), block (prevent hypothesis from proceeding to next stage), or escalate (pause all workflow, notify configured ethics officer or admin). The default for Tier 1 DURC is "escalate."

**False positive feedback loop**: Users can contest flags. Contested flags are logged and routed to a human reviewer (the project ethics board, when operational). The classifier is fine-tuned on contested flags to reduce false positives over time.

### B. Risk Tier Classification

The three-tier classification system (Phase 3.2) maps every research context to a risk level that determines governance requirements.

| Tier | Risk Level | Domains | Requirements |
|------|-----------|---------|-------------|
| Tier 1 | High | Virology, toxin research, AI safety, weapons, dual-use chemistry, human genetic engineering | Mandatory 24-hour cooling-off, external reviewer gate, human authorization required |
| Tier 2 | Medium | Clinical research without IRB, human subjects data, animal research, controlled substances | Mandatory human checkpoint, cannot proceed autonomously |
| Tier 3 | Low | Solar forecasting, materials science, ecology observation, mathematics, theoretical physics | Standard workflow, standard oversight |

**Domain auto-classification**: The EthicsAgent automatically classifies the research domain based on query content, literature context, and generated hypothesis content. Classification is not user-selectable (cannot self-declare as low risk), though manual override is possible with justification logged to provenance.

**Tier display**: The current risk tier is always displayed in the CLI/UI. Users always know which governance framework they are operating under. Tier is included in all outputs and exports.

### C. Mandatory Human Checkpoints

Human checkpoints are not optional features in openscire-hypothesis — they are architectural hard stops, analogous to physical interlocks in industrial safety systems.

**Before high-cost compute**: Any experimental design that exceeds the user's configured carbon or compute budget triggers a mandatory human checkpoint. The system presents the estimated cost, the expected benefit, and the uncertainty range. The workflow cannot proceed without explicit human sign-off.

**Before Tier 1/2 research**: Tier 1 research requires a 24-hour cooling-off period. The system holds the hypothesis, ethics report, and falsification report in a "pending review" state. After the cooling period, the human must explicitly authorize the workflow. Tier 2 research requires human notification and acknowledgment. The system will not proceed until the human acknowledges the risk and confirms.

**Before publication/dissemination**: Any output intended for publication (hypothesis reports, experimental protocols, findings summaries) must pass a human review checkpoint. The system presents its confidence scores, uncertainty flags, and verification gaps alongside the output. The human must sign off before export.

**Architectural nature**: These checkpoints are not UI that can be dismissed. They are enforced by the SupervisorAgent's state machine, which refuses transitions to the next state until the human handoff callback is received. There is no "auto-approve after timeout" — if the human does not respond, the workflow stays in the current state. The system has no override for its own ethics architecture [08_miscellaneous.md](../../critiques/08_miscellaneous.md#XV.54].

class AgentMessage(BaseModel):

**Hypothesis**:
```python
class Hypothesis(BaseModel):
    hypothesis_id: UUID
    claim: str                # the core claim

    confidence: float         # composite from UncertaintyQuantifier
    status: Literal["generated", "under_review", "surviving",
                    "falsified", "ethics_blocked", "retracted"]
    provenance: ProvenanceNode
```

### B. Agent Communication Protocol

**Message types**: Eight typed messages structure all agent communication:
- `Query` — agent requests information or analysis from another agent
- `Response` — agent provides requested information
- `Task` — SupervisorAgent assigns a subtask to a specialist agent
- `Result` — specialist agent returns task output
- `Review` — agent provides evaluation of another agent's output
- `Flag` — agent identifies a problem (ethical, factual, methodological)
- `Escalate` — agent requests human intervention
- `Log` — informational message (not actionable, persisted for provenance)

**AgentBus**: The message routing system that connects all agents:
- **Routing**: Messages are routed by recipient (direct), by thread_id (thread-scoped broadcast), or by message_type (topic subscription)
- **Delivery guarantees**: At-least-once delivery. Messages are persisted before delivery. Failures trigger retry (configurable, default 3)
- **Thread management**: All messages within a research context share a `thread_id`. Threads can be nested (sub-threads for specific falsification attempts), merged (when two threads converge), or archived (when research direction is abandoned)
- **Asynchronous handling**: Agents process messages as they arrive via non-blocking event loops. The AgentBus supports push (agent notified of new message) and pull (agent polls for messages) patterns

### C. Persistence

**Provenance DAG**: Every message, every state transition, every agent decision is stored in a directed acyclic graph (DAG) with immutable, append-only semantics. The DAG supports:
- Forward trace: given a hypothesis, find all agents, messages, and data that produced it
- Backward trace: given a claim, find what sources support it, who generated it, what alternatives were rejected
- Temporal query: "show all hypotheses generated between dates X and Y in domain Z"
- Conflict query: "show all agent disagreements that required human resolution"

**Storage backends**:
- **Default (local)**: SQLite via the openscire-core persistence layer. Portable, zero-configuration, suitable for individual researchers
- **Team/Enterprise**: PostgreSQL with JSONB for provenance data. Supports concurrent multi-user access, role-based query restrictions, and replication

**Export formats**:
- **RO-Crate**: Research Object Crate for FAIR-compliant sharing with other research infrastructure
- **W3C PROV**: Standard provenance interchange format for institutional repositories
- **JSON/CSV**: Flat exports for integration with external tools (Jupyter, RStudio, etc.)

## IX. Implementation Phases

openscire-hypothesis is built across three implementation phases, mapped to the existing OpenSciRe phase roadmap:

### Phase 6 — Multi-Agent Framework (MVP)

The 4-agent MVP is delivered in Phase 6, per [06_multi_agent.md](../../docs/phases/06_multi_agent.md):

- **Week 1-2 (6.1)**: Agent Communication Protocol — AgentMessage model, AgentBus routing, message persistence
- **Week 2-3 (6.2)**: SupervisorAgent — state machine, task queue, health monitoring, human handoff
- **Week 3-4 (6.3)**: LiteratureReviewAgent — query decomposition, multi-source dispatch, synthesis, gap identification, contradiction detection
- **Week 4-5 (6.4)**: FalsificationAgent — Popperian search, counter-example generation, confound identification, falsification report
- **Week 5 (6.5)**: EthicsAgent — EthicalFirewall integration, tier classification, dual-use flagging
- **Week 5 (6.6)**: NegativeResultRegistry — data model, storage, search, visualization, automated submission

**Phase 6 exit criteria**: End-to-end workflow completes: research question → literature review → hypothesis → falsification → ethics → negative result (if falsified) or report (if not falsified).

### Phase 7 — Hypothesis Generation & Experimental Design

Advanced capabilities are layered on in Phase 7, per [07_hypothesis_generation.md](../../docs/phases/07_hypothesis_generation.md):

- **Week 1 (7.1)**: HypothesisGenerator — literature-grounded synthesis, template library (PICO, FINER), multi-hypothesis generation, prioritization
- **Week 2 (7.2)**: SerendipityInjector — cross-domain analogy, contradiction-driven exploration, serendipity level parameter, novelty scoring
- **Week 3 (7.3)**: ExperimentalDesignAdvisor — experiment classification, control identification, power analysis, statistical methods, protocol templates
- **Week 4 (7.4)**: VerifiabilityAssessor with Cost-of-Test — falsifiability scoring, testability classification, cost estimation (time, money, ethics, access)
- **Week 4 (7.5)**: ProblemFormulationAdvisor — research question framing, assumption identification, alternative formulation generation

**Phase 7 exit criteria**: Can take a literature review, generate multiple hypotheses with serendipity, design experiments, estimate cost and verifiability, and assist with problem formulation.

### Phase 3 — Ethics Layer (Cross-Cutting)

All Phase 6 and Phase 7 agents integrate with the Phase 3 ethics layer [03_ethics_layer.md](../../docs/phases/03_ethics_layer.md):

- **3.1**: EthicalFirewall with DURC detection — keyword + embedding scanning, configurable actions, audit log
- **3.2**: Risk Tier Classification — three-tier system with mandatory human checkpoints
- **3.3**: DataSovereigntyChecker — consent verification, data origin classification, export restriction marking
- **3.5**: CarbonBudgetTracker — compute tracking, energy conversion, CO2e estimation, budget enforcement
- **3.6**: UncertaintyQuantifier — confidence scoring, contradiction detection, knowledge boundary flagging
- **3.7**: SourceGrounding — forced citation enforcement, source verification, unsupported claim flagging
- **3.8**: VerificationAsymmetryTracker — claim categorization, verification status tracking, verification path suggestion

**Deferred (Phase 7+)**:
- **NullHypothesisAgent**: Null model generation
- **HypothesisCriticAgent**: Structured multi-criteria evaluation
- **MethodologyAdvisorAgent**: Domain-specific experimental design
- **EvidenceValidatorAgent**: Independent evidence chain verification

## X. Competitive Context

### vs. Google Co-Scientist

| Dimension | Co-Scientist | openscire-hypothesis |
|---|---|---|
| Verification philosophy | Verification-first (confirm hypotheses) | Falsification-first (test to destruction) |
| Agent diversity | Homogeneous (same base model, same training data) | Heterogeneous (different models, objectives, contexts) |
| Negative results | Discarded (losing hypotheses invisible) | First-class citizens (NegativeResultRegistry) |
| Epistemic humility | None (equal confidence across domains) | Structural (UncertaintyQuantifier on every claim) |
| Source verification | Syntactic (does citation exist?) | Semantic (does citation support claim?) |
| Ethics | Absent (no dual-use firewall) | Structural (EthicsAgent as architectural gate) |
| Architecture | Cloud-native, API-dependent | Local-first, offline-capable, BYOK |
| Evaluation | Single hidden scoring metric | Transparent multi-criteria evaluation |
| Experiment design | No support | Full experimental design advisor |
| Problem formulation | Assumes user knows the question | Active assistance in question framing |
| Serendipity | None (within-paradigm optimization) | Cross-domain analogy, configurable exploration |
| Provenance | Minimal (clickable citations) | Full provenance DAG (W3C PROV / RO-Crate) |

### vs. Elicit / Scite / Consensus

These tools are literature synthesis and evidence search engines, not hypothesis generators. They answer questions by finding and summarizing existing literature. openscire-hypothesis includes this capability (via the LiteratureReviewAgent and Phase 4 engine) but extends far beyond it:

- **Generation, not just summarization**: openscire-hypothesis proposes new hypotheses that the literature does not yet contain, not just summaries of what the literature says
- **Falsification, not just support**: Where Elicit shows supporting and opposing evidence, openscire-hypothesis structurally tests hypotheses through adversarial multi-agent debate
- **Experimental design, not just analysis**: openscire-hypothesis designs the experiment to test the hypothesis, not just evaluates existing evidence
- **Negative result persistence, not just disappearance**: Failed hypotheses contribute to collective knowledge rather than vanishing

### vs. Traditional Hypothesis Generation Tools

Existing hypothesis generation tools (e.g., IBM Watson for Drug Discovery, BenevolentAI, various academic tools) typically operate as single-model systems that generate hypotheses from structured knowledge graphs or literature corpora. openscire-hypothesis differs in three fundamental ways:

- **Multi-agent adversarial architecture**: Genuine cognitive diversity through heterogeneous models and objective functions, not a single model generating multiple candidates
- **Epistemic Safety Layer**: Uncertainty quantification, source verification, verification asymmetry tracking, and confabulation detection at every stage — not just at output
- **Falsification-first epistemology**: Hypothesis generation is only half the system; the other half is dedicated falsification, which is architecturally equal in priority

## XI. Limitations Acknowledged

openscire-hypothesis is not a replacement for human scientific judgment. It is a tool designed to surface more of what the scientific process requires — evidence, uncertainty, alternatives, and failures — and to make those visible, auditable, and actionable. The following limitations are structural and acknowledged:

1. **NullHypothesisAgent and advanced agents are deferred to post-pilot**. The MVP deploys four core agents. The NullHypothesisAgent, HypothesisCriticAgent, MethodologyAdvisorAgent, and EvidenceValidatorAgent are specified and their interfaces defined but not implemented until Phase 7+. Hypothesis generation in the MVP relies on the LiteratureReviewAgent's implicit null-hypothesis awareness rather than a dedicated agent.

2. **Full interdisciplinary translation is limited by current LLM capabilities**. The SerendipityInjector's cross-domain analogy mechanism depends on the underlying models' ability to recognize structural isomorphisms between domains. Current LLMs can do this to a limited degree for closely related fields but struggle with maximally distant analogies. As models improve, the serendipity parameter will become more effective, but for the foreseeable future, the most surprising cross-domain connections will remain a human capacity.

3. **Physical/wet lab validation remains human responsibility**. The ExperimentalDesignAdvisor can propose protocols and estimate costs, but it cannot execute laboratory work, cannot verify that reagents are available in a specific lab, and cannot guarantee that a protocol will work in human hands. Physical validation remains the researcher's responsibility, and openscire-hypothesis is designed to support, not supplant, that work.

4. **The epistemic safety layer does not solve the verification regress problem**. The VerificationAsymmetryTracker makes the regress visible — it shows that the verifier shares the generator's architecture — but it cannot break the regress. Independent epistemic authority requires a differently-constructed cognitive system, which only a human (or a future AI with fundamentally different architecture) can provide. Every output from openscire-hypothesis should be treated as a candidate claim requiring human evaluation, not as a verified fact.

5. **The system inherits the biases of its underlying models and corpra**. Agent diversity and multi-source literature access mitigate but do not eliminate bias. If all configured models share a cultural or linguistic perspective (e.g., all are trained on English-language Western science), the system's outputs will reflect that perspective. The local-first architecture and BYOK model give users the freedom to choose different providers, but the responsibility for choosing diverse models rests with the user.

6. **The NegativeResultRegistry depends on user participation for completeness**. While automated submission from the FalsificationAgent captures a subset of negative results, the most valuable negative knowledge — from wet lab failures, from abandoned research directions, from the "not even tried" intuition of experienced scientists — can only be registered manually. The registry will always be incomplete, and its gaps reflect the same file-drawer problem it attempts to solve.

7. **There is no "Slow Science" mode in the MVP**. The system is designed with speed controls (Tier 1 cooling-off, configurable incubation periods, mandatory checkpoints), but a fully deliberative mode — one that explicitly slows down by default, encourages weeks of reflection, and structurally resists the speed-bias critique — is deferred to post-pilot. The MVP prioritizes falsification and ethics over temporal governance, and this is a limitation.

8. **The moral crumple zone remains structurally present**. Despite transparency and uncertainty display, the human user ultimately bears responsibility for decisions made using openscire-hypothesis. The system provides the evidence; the human provides the judgment. When judgment fails, the human bears the consequences. openscire-hypothesis cannot fix this asymmetry; it can only make it transparent.

9. **The material world interface is minimal**. openscire-hypothesis generates protocols, designs experiments, and estimates costs, but it cannot control laboratory equipment, ingest sensor data, or monitor physical experiments. The bridge between computational hypothesis generation and embodied scientific practice remains underdeveloped — a gap shared with every existing tool in this space.

10. **No governance framework for AI-generated discoveries**. openscire-hypothesis does not solve the regulatory vacuum around AI-generated drug targets, patentability of AI-discovered mechanisms, or liability for AI-proposed experiments. These require institutional and regulatory action beyond the scope of any software tool. openscire-hypothesis provides transparent provenance that can support regulatory submission, but it is the research ecosystem's responsibility to build the governance frameworks around it.
